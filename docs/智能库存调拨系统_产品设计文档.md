# 智能库存调拨系统产品设计文档

## 1. 项目名称

智能库存调拨系统  
英文名：Smart Inventory Transfer System  
简称：SITS

---

## 2. 项目定位

本系统面向多仓企业、供应链企业、制造企业，解决多个仓库之间库存不均衡的问题。

系统通过采集库存、销量、在途采购、仓库容量、物流成本、仓库优先级、区域需求等数据，自动识别：

- 哪些仓库库存不足
- 哪些仓库库存积压
- 哪些 SKU 存在断货风险
- 哪些 SKU 应该跨仓调拨
- 调拨多少数量最合适
- 从哪个仓调到哪个仓成本最低
- 调拨后是否会影响原仓销售

系统最终生成调拨建议，支持人工审批、自动执行、调拨跟踪、库存扣减、库存释放、异常补偿和 AI 辅助分析。

---

## 3. 项目价值

### 3.1 业务价值

传统库存管理依赖人工经验，容易出现：

1. 热销区域缺货，影响销售。
2. 冷门区域库存积压，占用资金。
3. 仓库之间信息不透明，调拨决策慢。
4. 调拨缺少数据依据，容易造成二次缺货。
5. 采购、销售、仓储、物流数据割裂。

智能库存调拨系统通过数据分析和规则决策，提高库存周转效率，降低缺货率和仓储成本。

### 3.2 技术价值

本项目适合作为企业级 Java 后端项目，核心技术点包括：

- 多仓库存模型
- 库存预占与释放
- 调拨审批流
- 状态机流转
- 规则引擎
- 定时任务
- RocketMQ 异步解耦
- Redis 缓存与分布式锁
- 幂等控制
- 异常补偿
- AI Copilot 辅助决策

## 5. 核心业务场景

### 5.1 库存不足场景

某 SKU 在华东仓销量较高，当前库存只够销售 2 天，系统判断存在断货风险。

系统自动分析其他仓库库存，发现华南仓库存充足，且未来 7 天销量较低，于是生成调拨建议：

```text
SKU：A001
调出仓：华南仓
调入仓：华东仓
建议调拨数量：300
预计支撑天数：10 天
预计物流成本：480 元
风险等级：高
```

### 5.2 库存积压场景

某 SKU 在华北仓库存 5000 件，但近 30 天销量只有 100 件，系统判断库存积压。

系统分析其他仓库需求，发现华东仓和华南仓销售较好，于是建议拆分调拨：

```text
华北仓 -> 华东仓：1000 件
华北仓 -> 华南仓：800 件
```

### 5.3 采购在途场景

某仓库库存不足，但已有采购单即将在 2 天后入库。

系统判断：

- 如果采购到货前不会断货，则不调拨。
- 如果采购到货前会断货，则只调拨短期补充数量。

### 5.4 人工审批场景

高价值调拨单需要主管审批。

例如：

```text
调拨金额 > 50000 元
或
调拨数量 > 1000 件
或
涉及核心仓库
```

系统自动进入审批流程。

### 5.5 调拨异常场景

调拨过程中可能出现：

- 出库失败
- 入库数量不一致
- 运输中丢失
- 调拨单超时未处理
- 库存扣减成功但 MQ 发送失败
- 出库成功但入库失败

系统需要通过状态机、补偿任务、操作日志保证最终一致性。

## 7.2 库存管理模块

### 7.2.1 仓库库存

每个 SKU 在每个仓库都有独立库存。

库存字段：

```text
available_stock      可用库存
locked_stock         锁定库存
in_transit_stock     调拨在途库存
safety_stock         安全库存
max_stock            最大库存
```

库存计算：

```text
实际库存 = 可用库存 + 锁定库存 + 在途库存
可售库存 = 可用库存
```

### 7.2.2 库存操作类型

库存变化来源包括：

- 采购入库
- 销售出库
- 调拨锁定
- 调拨出库
- 调拨入库
- 调拨取消释放
- 库存盘点调整

每次库存变化都必须记录库存流水。

### 7.2.3 库存流水

核心字段：

```text
flow_id
sku_id
warehouse_id
biz_type
biz_no
change_type
before_available
change_quantity
after_available
operator
create_time
```

## 7.3 销量统计模块

系统需要根据历史销量判断未来需求。

统计维度：

```text
SKU
仓库
日期
销量
销售金额
```

系统需要计算：

```text
近 7 天销量
近 14 天销量
近 30 天销量
日均销量
销量增长率
库存可支撑天数
```

库存可支撑天数：

```text
库存可支撑天数 = 当前可用库存 / 日均销量
```

如果日均销量为 0，则认为暂无缺货风险，但可能存在积压风险。

## 7.4 库存风险识别模块

系统定时扫描库存，识别库存风险。

### 7.4.1 缺货风险

```text
库存可支撑天数 <= 3 天：高风险
库存可支撑天数 <= 7 天：中风险
库存可支撑天数 <= 15 天：低风险
```

### 7.4.2 积压风险

```text
库存可支撑天数 >= 90 天：高积压
库存可支撑天数 >= 60 天：中积压
库存可支撑天数 >= 30 天：低积压
```

### 7.4.3 风险记录

```text
risk_id

warehouse_id
risk_type
risk_level
available_stock
avg_daily_sales
support_days
status
create_time
```

风险状态：

```text

PROCESSING   处理中
RESOLVED     已解决
IGNORED      已忽略
```

## 7.5 调拨建议模块

### 7.5.1 生成调拨建议的输入

```text
当前库存
安全库存
最大库存
近 7 天销量
近 30 天销量
采购在途数量
调拨在途数量
仓库距离
物流成本
仓库优先级
SKU 成本
风险等级
```

### 7.5.2 调入仓判断

调入仓一般来自缺货风险仓库。

```text
available_stock < safety_stock
或
support_days <= threshold_days
```

### 7.5.3 调出仓判断

调出仓必须满足：

```text
调出后库存 >= 安全库存
调出后可支撑天数 >= 最小保留天数
当前库存存在冗余
仓库状态正常
SKU 状态正常
```

### 7.5.4 建议调拨数量

基础公式：

```text
目标库存 = 日均销量 * 目标支撑天数
建议调拨数量 = 目标库存 - 当前可用库存 - 在途库存
```

同时限制：

```text
建议调拨数量 <= 调出仓可调拨数量
建议调拨数量 <= 调入仓最大库存 - 当前库存
建议调拨数量 >= 最小调拨数量
```

### 7.5.5 调拨评分

```text
score =
库存冗余度 * 0.35
+ 距离评分 * 0.25
+ 成本评分 * 0.20
+ 销量压力评分 * 0.10
+ 仓库优先级评分 * 0.10
```

系统选择评分最高的调拨方案。

### 7.5.6 调拨建议状态

```text
GENERATED     已生成
CONFIRMED     已确认
REJECTED      已拒绝
EXPIRED       已过期
CONVERTED     已转调拨单
```

## 7.6 调拨单模块

### 7.6.1 调拨单主流程

```text
创建调拨单
↓
锁定调出仓库存
↓
进入审批
↓
审批通过
↓
调出仓出库
↓
运输中
↓
调入仓入库
↓
调拨完成
```

### 7.6.2 调拨单状态

```text
CREATED          已创建
STOCK_LOCKED     库存已锁定
APPROVING        审批中
APPROVED         审批通过
REJECTED         审批拒绝
OUTBOUNDING      出库中
OUTBOUNDED       已出库
IN_TRANSIT       运输中
INBOUNDING       入库中
COMPLETED        已完成
CANCELLED        已取消
FAILED           调拨失败
```

### 7.6.3 状态流转

```text
CREATED
  ↓ lock_stock
STOCK_LOCKED
  ↓ submit_approval
APPROVING
  ↓ approve
APPROVED
  ↓ outbound
OUTBOUNDING
  ↓ outbound_success
OUTBOUNDED
  ↓ ship
IN_TRANSIT
  ↓ inbound
INBOUNDING
  ↓ inbound_success
COMPLETED
```

异常流转：

```text
APPROVING -> REJECTED
CREATED -> CANCELLED
STOCK_LOCKED -> CANCELLED
OUTBOUNDING -> FAILED
INBOUNDING -> FAILED
```

### 7.6.4 库存处理

创建调拨单后，先锁定调出仓库存：

```text
available_stock -= quantity
locked_stock += quantity
```

出库成功后：

```text
locked_stock -= quantity
in_transit_stock += quantity
```

入库成功后：

调出仓：

```text
in_transit_stock -= quantity
```

调入仓：

```text
available_stock += quantity
```

取消调拨时：

```text
locked_stock -= quantity
available_stock += quantity
```

```text

```

## 7.9 AI Copilot 模块

AI Copilot 作为智能分析助手，不直接修改核心业务数据，只提供分析、解释和建议。

### 7.9.1 AI 可以做

```text
解释为什么生成该调拨建议
分析库存不足原因
总结某个 SKU 的库存风险
给出调拨方案对比
生成审批意见摘要
生成运营日报
```

### 7.9.2 AI 不可以做

```text
直接扣减库存
直接创建正式调拨单
直接审批通过
直接修改库存
```

所有核心动作必须由业务系统规则和人工确认完成。

### 7.9.3 Agent 设计

```text
库存风险分析 Agent
调拨方案生成 Agent
物流成本分析 Agent
审批摘要 Agent
异常诊断 Agent
```

### 7.9.4 AI 工具接口

AI 不直接查数据库，而是通过系统提供的只读工具接口获取数据：

```text
查询 SKU 库存
查询销量趋势
```

## 12. 状态机设计

### 12.1 调拨单状态

```java
public enum TransferOrderStatus {
    CREATED,
    STOCK_LOCKED,
    APPROVING,
    APPROVED,
    REJECTED,
    OUTBOUNDING,
    OUTBOUNDED,
    IN_TRANSIT,
    INBOUNDING,
    COMPLETED,
    CANCELLED,
    FAILED
}
```

### 12.2 调拨单事件

```java
public enum TransferOrderEvent {
    LOCK_STOCK,
    SUBMIT_APPROVAL,
    APPROVE,
    REJECT,
    START_OUTBOUND,
    OUTBOUND_SUCCESS,
    START_SHIP,
    START_INBOUND,
    INBOUND_SUCCESS,
    CANCEL,
    FAIL
}
```

### 12.3 状态流转规则

```text
CREATED + LOCK_STOCK -> STOCK_LOCKED
STOCK_LOCKED + SUBMIT_APPROVAL -> APPROVING
APPROVING + APPROVE -> APPROVED
APPROVING + REJECT -> REJECTED
APPROVED + START_OUTBOUND -> OUTBOUNDING
OUTBOUNDING + OUTBOUND_SUCCESS -> OUTBOUNDED
OUTBOUNDED + START_SHIP -> IN_TRANSIT
IN_TRANSIT + START_INBOUND -> INBOUNDING
INBOUNDING + INBOUND_SUCCESS -> COMPLETED
CREATED + CANCEL -> CANCELLED
STOCK_LOCKED + CANCEL -> CANCELLED
任意处理中状态 + FAIL -> FAILED
```

## 17. 异常补偿设计

### 17.1 库存锁定成功，调拨单状态更新失败

补偿策略：

```text
根据 transfer_no 查询库存流水
如果库存已锁定但调拨单仍是 CREATED
则重试状态更新为 STOCK_LOCKED
```

### 17.2 出库成功，MQ 发送失败

补偿策略：

```text
创建补偿任务
定时扫描 OUTBOUNDED 状态调拨单
重新发送 TransferOutboundSuccessEvent
```

### 17.3 入库失败

补偿策略：

```text
调拨单进入 FAILED
创建异常记录
允许人工重试入库
必要时执行反向调拨或库存调整
```

## 24. 简历描述示例

### 项目描述

智能库存调拨系统面向多仓企业库存管理场景，围绕库存不足、库存积压、跨仓调拨、审批流转和异常补偿等核心问题，构建了一套企业级库存调拨平台。系统通过定时任务分析仓库库存、历史销量、在途库存和安全库存，自动识别缺货/积压风险，生成调拨建议，并通过状态机驱动调拨单从库存锁定、审批、出库、运输到入库完成的全流程流转。同时引入 AI Copilot，对调拨建议进行原因解释、风险分析和方案对比，提高运营决策效率。

### 个人职责

1. 负责库存中心设计，实现多仓 SKU 库存模型，支持可用库存、锁定库存、在途库存的独立管理。
2. 设计库存流水和库存幂等机制，通过业务单号和操作类型唯一约束，防止 MQ 重复消费导致库存重复扣减。
3. 基于 Spring StateMachine 实现调拨单状态流转，保证调拨单在审批、出库、入库等节点的状态合法性。
4. 设计库存风险扫描任务，基于近 7 天、近 30 天销量计算库存可支撑天数，自动识别缺货和积压风险。
5. 实现调拨建议生成算法，综合库存冗余度、销量压力、物流成本、仓库优先级等因素计算调拨评分。
6. 使用 Redis 分布式锁解决同一 SKU 在同一仓库下并发调拨导致的库存竞争问题。
7. 使用 RocketMQ 解耦风险识别、调拨建议生成、审批通知、库存流水记录等流程，并设计消费幂等表保证消息可靠处理。
8. 设计补偿任务机制，对库存锁定失败、出库异常、入库异常、MQ 发送失败等情况进行自动重试和人工介入。
9. 接入 AI Copilot，通过工具调用查询库存、销量、调拨单等数据，实现调拨建议解释、风险分析和日报生成。

### 项目亮点

1. 通过状态机统一管理调拨单生命周期，避免业务代码中大量 if-else 判断，提高状态流转的可维护性。
2. 通过库存流水、乐观锁、分布式锁、MQ 幂等表，保证高并发调拨场景下库存数据最终一致。
3. 通过库存可支撑天数模型和调拨评分算法，实现缺货风险识别和跨仓调拨建议自动生成。
4. 通过补偿任务兜底处理分布式链路异常，提升调拨流程的可靠性。
5. 引入 AI Copilot 作为供应链分析助手，实现库存风险解释、调拨方案对比和运营日报生成。
